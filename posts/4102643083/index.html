CMDB 后端技术决策和架构记录

核心架构

资源类型模式

每种资源类型遵循统一的三层架构：

两类节点模式

  • 一类节点 (CTYUN)：通过 CTYunCommonApiClient（CTECS OpenAPI SDK）获取全量资源
  • 二类节点 (CTYUN_LEVEL2)：通过 CTYunLevel2HttpClient（EOP-Authorization 签名）或 cn.ctyun.sdk SDK 调用各服务 API

已支持的资源类型

  • VM（弹性云主机）：一类(CTECS SDK) + 二类(ECS区域API)
  • MySQL（关系数据库）：一类(RDS2 SDK)
  • Redis（分布式缓存）：一类(DCS2 SDK)
  • ELB（弹性负载均衡）：一类(CTELB SDK)
  • 云硬盘 EVS（新建）：一类(EVS API，端点待验证)

关键技术决策

1. 多租户插件处理方案

  • 问题：MyBatis-Plus 的 DandelionTenantInterceptor 自动给所有查询加 tenant_id = ?
  • 定时任务：无登录上下文，SecurityUtils.getTenantId() 抛异常
  • 方案：使用 InterceptorIgnoreHelper.handle(IgnoreStrategy.builder().tenantLine(true).build()) 临时跳过
  • 规范：必须用 try-finally 确保 clearIgnoreStrategy() 执行，否则后续所有查询跳过租户过滤

2. 资源统计查询方案（重要）

  • JdbcTemplate（已被否决）：直写 SQL 绕过插件，改动太大
  • MyBatis Mapper + InterceptorIgnoreHelper（选定的方案）：
    • 使用框架标准 API 临时绕过租户过滤
    • 保持与项目整体一致的数据访问层
    • try-finally 确保清理

3. 二类节点同步改为数据库驱动

  • 原方案：硬编码 AK/SK 在 Controller 中
  • 新方案：从 cmdb_asset_account 读取所有 CTYUN_LEVEL2 类型账号，遍历同步
  • 循环依赖解决：CTYunLevel2EcsServiceImpl 改用 CTYunAssetAccountMapper 直查，绕过 CTYAssetAccountService

4. VM 去重策略

  • 三个一类账号（长新云/多云/我的）同步完全重叠的 20 个 VM
  • Mapper XML 中使用子查询去重：
  • 不传 batchId 时去重生效

Flyway 迁移说明

  • 迁移文件位置：dandelion-admin/src/main/resources/db/migration/
  • 手动执行 SQL 后必须在 flyway_schema_history 表同步记录
  • checksum 必须与文件内容匹配，否则 Flyway validate 失败

已知问题和待办

  1. 云硬盘 EVS API 端点待验证：猜测 https://evs-global.ctapi.ctyun.cn，实际端点需要从控制台 F12 确认
  2. 资源概览实时计数已移除：因 15+ 区域 API 调用超时，暂时只显示 DB 已同步的 4 种资源计数
  3. eventEdit 关联云资源权限：临时移除了 $auth.hasPermi 校验，后续需要补回
  4. Volume/VPC/NAT 等列表页未实现：目前只有计数，没有列表/详情页面
